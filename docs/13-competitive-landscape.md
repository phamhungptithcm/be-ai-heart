# Competitive Landscape

## Summary

The market is real. `be-ai-heart` is entering an active category, not an empty one.

There are already multiple products positioning around codebase context, indexing, semantic retrieval, or graph-aware reasoning for AI-assisted engineering.

That is good news because it validates demand.

It also means `be-ai-heart` should not position itself as “AI that understands your repo” in a generic way. That message is already occupied.

## Main Adjacent Players

### Sourcegraph Cody

Sourcegraph documents that Cody uses keyword search, Sourcegraph Search, and a code graph to retrieve context from local and remote codebases.

Positioning takeaway:

- strong enterprise code search and codebase context
- strong remote repository awareness
- already established in enterprise environments

### Cursor

Cursor documents automatic codebase indexing, incremental updates, multi-root workspaces, and PR search that brings repository history into context.

Positioning takeaway:

- strong IDE-native developer adoption
- good index-based repo understanding
- less explicitly positioned as a separate enterprise context operating layer

### Augment Code

Augment markets a “Context Engine” with a live understanding of code across repos, services, history, docs, tickets, and design decisions, and exposes it via MCP and SDK.

Positioning takeaway:

- very close to the core thesis of `be-ai-heart`
- strong competitive pressure on multi-repo context and agent grounding
- important proof that a context-engine category can be productized

### Qodo Context Engine

Qodo positions its context engine as deep codebase intelligence with multi-repo indexing, organizational knowledge, and agentic reasoning across the SDLC, including API and MCP access.

Positioning takeaway:

- closest overlap on “turn code into knowledge”
- strong enterprise and governance angle
- demonstrates that buyers care about context as infrastructure, not only as IDE convenience

### Greptile

Greptile documents a graph-based codebase context approach for review, describing repository-wide graphs of functions, classes, imports, and dependencies used during pull request analysis.

Positioning takeaway:

- strongest adjacency on graph-based reasoning
- narrower use case around review rather than full project memory

## Strategic Assessment

The idea is strong, but not unique.

The opportunity is not in inventing “context for AI coding” from zero. The opportunity is in choosing a sharper product wedge and executing better on one of the following:

1. `Project memory across code and documents`
2. `MCP-native context infrastructure for any agent`
3. `ROI and token-savings proof for enterprises`
4. `Governance and safe reusable AI work in engineering teams`

## Best Differentiation For `be-ai-heart`

The strongest defensible position from your original idea is:

`be-ai-heart` is the project memory and governance layer for AI coding, combining code graph, document memory, architecture policy, and benchmark-based ROI proof.

That is narrower and more defensible than simply saying “we index your repo.”

## Risks

- If positioned as generic codebase context, `be-ai-heart` will look like a weaker version of larger players.
- If positioned only as token savings, it may look tactical and easy to replace.
- If positioned only as review or search, it will lose the broader system-memory advantage.

## Recommended Positioning

Lead with:

- stable project memory
- code plus document context
- reusable architecture-aware context packs
- benchmark-backed savings and safety
