# Keep local and portal memory aligned

The CLI is the source of truth for repository memory. The portal should reflect reviewed artifacts from the repo, not pretend hosted data is current when the local scan is stale.

## 1. Scan source and documents

Run `heart scan` after meaningful repository changes. The scan refreshes code graph, document memory, policy posture, diagram inputs, and benchmark readiness.

```bash
heart doctor
heart scan
heart overview
```

## 2. Generate diagrams for review

Diagrams are review artifacts. Use them to make repo structure easier to inspect before publishing summaries into the portal.

```bash
heart diagram generate high-level
heart diagram generate sequence
```

## 3. Sync safe summaries

Sync publishes reviewed repository artifacts: profile, diagrams, docs freshness, context pack metadata, policies, and benchmark reports where supported.

Raw private source should not be assumed to leave the local lane by default. Use `heart sync setup` for the first activation because it publishes profile, docs, and a starter hosted context pack together.

```bash
heart login
heart sync setup --slug acme-billing --task "prepare first implementation"
```

## 4. Verify portal truth

The portal repo page should show:

- last CLI sync
- scan status
- graph health
- docs/spec freshness
- generated diagrams
- context pack history
- benchmark evidence
- policy warnings

If those values are stale, rerun the local loop and sync again.

Portal chat uses the same synced artifacts. The workbench stream endpoint can attach repo, docs, graph, domain pack, and selected context pack metadata, but it still cannot run arbitrary shell commands from the browser.
